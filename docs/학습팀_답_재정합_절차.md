## 3. 갱신 절차 (Step-by-Step)

### Step 0 — 답 도착 분류
학습팀에서 받은 정보를 다음 4종 중 하나(또는 여러 개)로 분류:

- ☐ **A. 클래스 ID 순서·명칭 통보** (#2·#5 핵심) → §3-A 진행
- ☐ **B. 박리-도장 흡수 결정** (#1) → §3-B 진행
- ☐ **C. 실제 모델 파일(`.pth`/`.tflite`) 전달** (#5) → §3-C 진행
- ☐ **D. 일정·tflite 크기 실측 통보** (#6) → §3-D 진행

복수 도착 시 A → B → C → D 순서로 진행.

---

### Step 1-A. 클래스 ID 순서·명칭 갱신 (5곳 일괄)

학습팀의 metadata.json 또는 인덱스-라벨 매핑 표를 받은 뒤:

#### 1) JSONB 형식 변환
```
{"0":"<클래스명_0>","1":"<클래스명_1>", ..., "29":"<클래스명_29>"}
```
(공백 없음, 따옴표는 큰따옴표, 키는 문자열)

#### 2) ★ 5곳 + α1·α2 갱신

| 위치 | 변경 형식 |
|---|---|
| ① `backend/sql/002_seed.sql` L33·L34 | JSONB 문자열 양쪽 작은따옴표 안에 위 JSONB 그대로 |
| ② `tutorial/02_백엔드_기반_구축.md` L422·L423 | 위와 동일 |
| ③ `tutorial/09` `_CLASSES` | Python 리스트로 변환: `_CLASSES = ["균열-도장", ..., "폼스프레이양품-우레탄폼"]` |
| ④ `tutorial/14` `_CLASSES` | 위와 동일 Python 리스트 |
| ⑤ `inspection_app/SINGLE_MODEL_GUIDE.md` `_unifiedLabels` | Dart 리스트: `static const List<String> _unifiedLabels = ['균열-도장', ..., '폼스프레이양품-우레탄폼'];` |
| α1 `tutorial/13` 두 곳 (`_unifiedLabels` 본문 + 정리본) | Dart 리스트 동일 |
| α2 `tutorial/11` `defectChoices` | Dart 리스트 동일 (그룹 헤더 주석 유지) |

> 💡 동일 데이터를 4개 언어(JSONB·SQL string·Python·Dart) 로 변환하므로, **클래스 30개 명칭은 1줄에 모아두고 형식만 변환**하는 게 안전.

---

### Step 1-B. 박리-도장 흡수 결정 (드물게 발생)

현재 사용자 결정 (`명세서/계획 변경 내용.md` §8 #1 / §9) = **폐기** 확정. 학습팀이 폐기 외 결정(예: `도막떨어짐-도장` 또는 `도막분리-도장`으로 흡수)을 통보하면:

- α5 `tutorial/14` L293 매뉴얼 ETL 청크의 `박리-도장 발견 시: ...` 콘텐츠를 **흡수 클래스의 매뉴얼**로 변환 (예: `도막떨어짐-도장 발견 시: 박리된 부분과 그 주변 30mm까지 ...` 등)
- 위반 검출용 `scripts/check_single_model_integrity.ps1` 의 검증 항목은 영향 없음 (박리-도장은 어차피 30클래스에 미포함이므로 grep 잡힘 0)
- 사용자 §C 확정 사항 변경이 필요한 경우 별도 의사결정 회의

---

### Step 1-C. 실제 모델 파일 전달

학습팀이 `best_model_v5_datamatch_full.pth`·`.tflite` 가중치 파일과 검증 정확도 보고서를 전달하면:

1. **파일 배치**:
   - `inspection_app/assets/models/best_model_v5_datamatch_full.tflite` 에 `.tflite` 복사
   - `backend/models/best_model_v5_datamatch_full.pth` 에 `.pth` 복사 (`tutorial/14` §A-2 참조)
2. **SHA-256 계산** + `모델_레지스트리.파일_해시` 갱신:
   ```powershell
   (Get-FileHash inspection_app\assets\models\best_model_v5_datamatch_full.tflite -Algorithm SHA256).Hash
   ```
   ```sql
   UPDATE 모델_레지스트리 SET 파일_해시 = 'sha256:실측해시' WHERE 활성_여부 = true;
   ```
3. **Firebase ML Custom Models** 에 `.tflite` 게시 (`tutorial/16` §3-1)
4. **OTA-002** 로 새 활성 모델 등록 (`tutorial/16` §4-3)

## 5. 자주 발생하는 오류

### "한 곳을 빼먹어서 회귀 검증 fail"
- 본 절차서 §2 표의 ★ 5곳을 **체크리스트로 출력해서 손으로 ✓** 후 진행 권장
- `scripts/check_single_model_integrity.ps1` 의 Check 5 가 30클래스 4자 일치를 자동 비교하므로, fail 발생 시 어느 파일이 안 됐는지 출력됨

### "JSONB 따옴표 깨짐"
- SQL 안 작은따옴표 사이 JSONB 의 큰따옴표는 그대로 가능. 단, JSON 키-값에 작은따옴표가 들어가면 SQL 파싱 깨짐 → 30클래스 명칭에 작은따옴표 들어가지 않게 학습팀에 사전 요청

### "Python 리스트 ↔ Dart 리스트 변환 오타"
- Python 은 `[...]`·작은/큰따옴표 모두 허용. Dart 는 `[...]`·작은따옴표 권장 + 끝에 세미콜론(`;`).
- 변환 스크립트가 있으면 좋지만, 30개 → 30개 변환은 손으로도 5분 내. 다만 마지막 항목 trailing comma 확인

### "Firebase 새 .tflite 업로드 후 OTA 갱신 안 됨"
- `tutorial/16` §자주 오류 "SHA-256 불일치" / "model not found" 케이스 참조
- DB `파일_해시` 와 실제 파일 해시 일치하는지 PowerShell `Get-FileHash` 로 재확인
