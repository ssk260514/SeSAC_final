## 1. 시스템 구성도 (개략)

```
[검사원 단말 (Android, MDM 설치)]
        │  HTTPS + JWT(30분/refresh 7일)
        ▼
[FastAPI 백엔드 (AWS EC2 g4dn.xlarge, Docker)]
        │
        ├─→ [PostgreSQL + pgvector]   ← 매뉴얼 임베딩 1024차원
        ├─→ [단일 통합 모델 .pth (30클래스)]  ← MobileNetV3-Large + Focal Loss
        ├─→ [vLLM + Qwen 2.5 7B AWQ]  ← RAG 조치 가이드 생성
        └─→ [AWS S3]                   ← 검사 이미지 + 양품 샘플 10%

[Firebase ML Custom Models (별도 Google 계정)]
        ↓ best_model_v5_datamatch_full.tflite (모델 파일만)
[검사원 단말 — OTA 다운로드]

[사내 MDM (Microsoft Intune / VMware Workspace ONE / Samsung Knox Manage)]
        ↓ APK 자동 푸시
[검사원 단말 — Required 정책 자동 설치]

[GitHub Actions]
        ├─→ Flutter APK 자동 빌드·서명 (.github/workflows/flutter-build.yml)
        └─→ Backend Docker 이미지 → GHCR
```

## 2. 운영자 절차 (자주 쓰는 작업)

### 2-1. 새 모델 버전 배포 (재학습 완료 시)
1. Firebase Console → Custom 모델 `best_model_v5_datamatch_full` 새 `.tflite` 교체 게시
2. SHA-256 계산:
   ```powershell
   (Get-FileHash best_model_v5_datamatch_full.tflite -Algorithm SHA256).Hash
   ```
3. DB 모델_레지스트리 INSERT (활성_여부=false):
   ```sql
   INSERT INTO 모델_레지스트리 (공정_ID, 모델_버전, 모델_유형, 파일_경로, 파일_해시, 클래스_라벨, 활성_여부)
   VALUES (NULL, 'v2', 'TFLITE', 'firebase://best_model_v5_datamatch_full',
           'sha256:계산한해시', '{"0":"균열-도장", ..., "29":"폼스프레이양품-우레탄폼"}', false);
   ```
4. OTA-002 로 활성화:
   ```bash
   curl -X PATCH https://api.example.com/api/admin/models/{새_model_id}/activate \
        -H "X-Admin-API-Key: $ADMIN_API_KEY"
   ```
5. 다음 앱 시작 시 검사원 단말에 자동 다운로드·SHA-256 검증·교체

> 상세: [`tutorial/16_3주차_모델OTA_Firebase.md`](../tutorial/16_3주차_모델OTA_Firebase.md) §4-3

### 2-2. MDM APK 업데이트
1. GitHub Actions `flutter-build` 워크플로 → Artifacts 에서 `app-release-{sha}.apk` 다운로드
2. MDM 콘솔 → 기존 앱 항목 → "Update" → 새 APK 업로드 (`versionCode` 자동 증가 확인)
3. 배포 그룹 (검사원 단말 그룹) → Required 정책 유지 → 단말 충전 + Wi-Fi 시 자동 설치
4. MDM 콘솔에서 디바이스별 설치 성공률 모니터링 (목표 95%+)

> 상세: [`tutorial/19_4주차_MDM배포_JWT안정화.md`](../tutorial/19_4주차_MDM배포_JWT안정화.md) §3

### 2-3. 운영 지표 일일 확인
```sql
-- 매일 아침
SELECT * FROM v_class_distribution_24h;
SELECT * FROM v_pass_fail_ratio_24h;
SELECT * FROM v_latency_24h;

-- 매일 오후 (수정률)
SELECT * FROM v_inspector_correction_rate ORDER BY 수정률_퍼센트 DESC NULLS LAST LIMIT 10;
```

또는 관리자 API:
```bash
curl -H "X-Admin-API-Key: $ADMIN_API_KEY" https://api.example.com/api/admin/monitoring/summary
```

> 상세: [`tutorial/20_4주차_모니터링과_파일럿.md`](../tutorial/20_4주차_모니터링과_파일럿.md) §3

### 2-4. 회귀 정합 검증 (코드 변경 후)
```powershell
# Windows
.\scripts\check_single_model_integrity.ps1
```
```bash
# macOS/Linux
./scripts/check_single_model_integrity.sh
```
→ exit 0 (위반 0건)이면 단일 모델 정합 유지. GitHub Actions 워크플로 step 으로 끼워넣어 자동화 가능.

## 3. 비밀번호·키 관리 (보안 분리)

| 자산 | 보관 위치 | 갱신 주기 | 비고 |
|---|---|---|---|
| `JWT_SECRET_KEY` | EC2 `.env` (+ AWS Secrets Manager 백업) | 분기 | 갱신 시 모든 검사원 강제 재로그인 |
| `ADMIN_API_KEY` | EC2 `.env` (+ AWS Secrets Manager 백업) | 분기 | OTA-002 활성화 API 인증용 |
| `upload-keystore.jks` | 별도 Vault 또는 1Password | 영구 (잃으면 앱 재배포 불가) | GitHub Secrets `KEYSTORE_BASE64` 동기화 |
| Keystore 비밀번호 | 1Password 공유 vault | 영구 | GitHub Secrets `KEYSTORE_PASSWORD`/`KEY_PASSWORD` |
| Firebase Service Account | Google Cloud Console (별도 계정) | 분기 | 회사 메인 계정과 분리 — 영업비밀 보호 |
| AWS IAM Access Key | AWS Secrets Manager | 분기 | S3 `lng-inspection-samples` 권한만 |
| DB 비밀번호 | EC2 `.env` (+ AWS Secrets Manager) | 분기 | PostgreSQL `검사원` 테이블의 `비밀번호_해시`와 별개 |

> ⚠️ `.env` 파일은 git 커밋 금지. `upload-keystore.jks` 도 git 절대 커밋 금지 (`.gitignore` 확인).

## 6. 단일 모델 정합 핵심 원칙 (인수인계 후에도 유지)

| 원칙 | 위반 시 발생 문제 |
|---|---|
| 모델 파일명 `best_model_v5_datamatch_full` 고정 | OTA 다운로드 실패 (Firebase 모델명 mismatch) |
| 30클래스 (불량 19 + 양품 11) 고정 | TFLite 출력 인덱스와 라벨 mismatch → 분류 오류 |
| 양품 판별 = 클래스명에 `"양품"` 포함 | 검사_피드백.품질_여부 BOOLEAN 잘못 도출 |
| OTA-001 `?process_id` 파라미터 없음 | 단일 모델 정합 위반 (API 계약 불일치) |
| 모델_레지스트리 `WHERE 모델_유형=` 만 (공정_ID 조건 없음) | OTA-002 트랜잭션 오작동 |
| RAG `WHERE 매뉴얼.공정_ID` 유지 | RAG 매뉴얼 범위 무한정 확장 → 응답 시간 폭증 |
| 신뢰도 임계값 전역 0.85 / 0.70 (공정별 차등 없음) | INFER-002 분기 오작동 |

→ 코드/문서 변경 후 반드시 `scripts/check_single_model_integrity.ps1` 통과 확인
