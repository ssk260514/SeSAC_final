## 1. 왜 모니터링 + 파일럿인가

### 1-1. 모니터링
- 모델은 실제 환경에 배포되면 학습 시와 다른 분포의 데이터를 만남(조명·앵글·계절·새 결함 유형) → **성능 저하(drift)** 발생
- 30클래스 중 어느 클래스에서 신뢰도가 낮은지·검사원 수정률이 높은지 추적해야 **재학습 우선순위** 결정 가능
- GPU 메모리 12GB 이하 유지(`아키텍처_명세서.md` Part 6 4주차 끝)도 모니터링 대상

### 1-2. 파일럿
- 5~10명 검사원이 실제 현장에서 1주일 사용 → 명세서로는 잡히지 않는 UX 문제(터치 영역·라벨 가독성·통신 음영 분포)를 발견
- 검사원 수정 데이터(`검사_피드백`)가 누적되어 **재학습 데이터 + 모델 신뢰도 보정**의 근거가 됨

```
파일럿 시작
    │
    ├─ 검사원 5~10명 단말기 MDM 배포(19번)
    ├─ 실시간 대시보드로 정확도·응답시간·클래스별 분포 추적
    ├─ 일일 피드백 수집(슬랙 채널 + 인앱 의견 메모)
    └─ 1주 후 회고 → 재학습 트리거 결정 → 인수인계
```

## 3. 1단계 — 간단한 로그 대시보드 (Stage 1)

복잡한 도구 대신 **PostgreSQL 쿼리 + Grafana** 또는 **간단한 FastAPI + Chart.js** 로 시작합니다.

### 3-1. 핵심 지표 SQL

`backend/sql/004_monitoring_views.sql` (신규):

```sql
-- 클래스별 추론 분포 (최근 24시간)
CREATE OR REPLACE VIEW v_class_distribution_24h AS
SELECT
    결함_유형,
    COUNT(*) AS 건수,
    AVG(신뢰도_점수) AS 평균_신뢰도,
    SUM(CASE WHEN 사람_재확인_필요 THEN 1 ELSE 0 END) AS 재확인_건수
FROM 검사_결과
WHERE 생성_일시 > NOW() - INTERVAL '24 hours'
  AND 대표_여부 = true
GROUP BY 결함_유형
ORDER BY 건수 DESC;

-- 검사원 수정률 (모델 정확도 proxy) — 단일 모델 30클래스 전체
CREATE OR REPLACE VIEW v_inspector_correction_rate AS
SELECT
    r.결함_유형 AS 원본_클래스,
    COUNT(f.피드백_ID) AS 피드백_수,
    SUM(CASE WHEN f.검사원_수정_여부 THEN 1 ELSE 0 END) AS 수정_수,
    ROUND(100.0 * SUM(CASE WHEN f.검사원_수정_여부 THEN 1 ELSE 0 END) / NULLIF(COUNT(f.피드백_ID), 0), 1) AS 수정률_퍼센트
FROM 검사_결과 r
JOIN 검사_피드백 f ON f.결과_ID = r.결과_ID
WHERE r.생성_일시 > NOW() - INTERVAL '7 days'
  AND r.대표_여부 = true
GROUP BY r.결함_유형
ORDER BY 수정률_퍼센트 DESC NULLS LAST;

-- 응답 시간 분포
CREATE OR REPLACE VIEW v_latency_24h AS
SELECT
    추론_위치,
    percentile_cont(0.50) WITHIN GROUP (ORDER BY 추론_지연_ms) AS p50_ms,
    percentile_cont(0.95) WITHIN GROUP (ORDER BY 추론_지연_ms) AS p95_ms,
    percentile_cont(0.99) WITHIN GROUP (ORDER BY 추론_지연_ms) AS p99_ms,
    COUNT(*) AS 건수
FROM 검사_결과
WHERE 생성_일시 > NOW() - INTERVAL '24 hours'
GROUP BY 추론_위치;

-- 양품/불량 비율 (단일 모델 — '양품' 포함 여부)
CREATE OR REPLACE VIEW v_pass_fail_ratio_24h AS
SELECT
    CASE WHEN 결함_유형 LIKE '%양품%' THEN '양품' ELSE '불량' END AS 판정,
    COUNT(*) AS 건수,
    AVG(신뢰도_점수) AS 평균_신뢰도
FROM 검사_결과
WHERE 생성_일시 > NOW() - INTERVAL '24 hours'
  AND 대표_여부 = true
GROUP BY 1;
```

> 💡 **수정률 = 모델 정확도 proxy**: 검사원이 AI 판정값을 바꾸지 않고 그대로 저장하면 "모델이 맞췄다"는 강한 신호. 클래스별 수정률이 30% 넘으면 그 클래스는 재학습 후보입니다. 단일 모델 30클래스 모두 같은 기준으로 평가합니다 (공정별 차등 없음).

### 3-2. FastAPI 모니터링 엔드포인트

`backend/app/api/monitoring.py` (신규):

```python
from fastapi import APIRouter, Depends, Header, HTTPException
from sqlalchemy.ext.asyncio import AsyncSession
from sqlalchemy import text

from app.core.db import get_db
from app.core.config import settings

router = APIRouter()


@router.get("/admin/monitoring/summary")
async def monitoring_summary(
    x_admin_api_key: str = Header(default=""),
    db: AsyncSession = Depends(get_db),
):
    """관리자용 단일 모델 운영 지표 요약."""
    if x_admin_api_key != settings.ADMIN_API_KEY:
        raise HTTPException(status_code=401, detail={"error": "ADMIN_AUTH_FAILED"})

    classes = (await db.execute(text("SELECT * FROM v_class_distribution_24h"))).all()
    corrections = (await db.execute(text("SELECT * FROM v_inspector_correction_rate"))).all()
    latency = (await db.execute(text("SELECT * FROM v_latency_24h"))).all()
    pass_fail = (await db.execute(text("SELECT * FROM v_pass_fail_ratio_24h"))).all()

    return {
        "class_distribution_24h": [dict(r._mapping) for r in classes],
        "correction_rate_7d":     [dict(r._mapping) for r in corrections],
        "latency_24h":            [dict(r._mapping) for r in latency],
        "pass_fail_24h":          [dict(r._mapping) for r in pass_fail],
        "model": "best_model_v5_datamatch_full",
        "class_count": 30,
    }
```

이 단일 엔드포인트로 단순한 HTML 대시보드를 그릴 수 있습니다(Chart.js 등).

## 4. 2단계 — Evidently AI (선택, 더 정교한 추적)

대시보드가 안정되면 [Evidently AI](https://www.evidentlyai.com/) 같은 ML 모니터링 도구로 확장:

- **Data Drift**: 학습 데이터 분포 vs 운영 데이터 분포 변화 자동 감지
- **Class Imbalance**: 30클래스 중 어느 클래스가 운영에서 거의 안 들어오는지(데이터 부족 신호)
- **Confidence Drift**: 신뢰도 평균이 시간에 따라 떨어지면 drift 신호

```python
# backend/ml/monitoring/evidently_report.py (선택)
from evidently.report import Report
from evidently.metric_preset import DataDriftPreset, ClassificationPreset
import pandas as pd

reference = pd.read_csv("data/train_distribution.csv")  # 학습 데이터 라벨 분포
current = pd.read_sql("SELECT 결함_유형, 신뢰도_점수 FROM 검사_결과 "
                       "WHERE 생성_일시 > NOW() - INTERVAL '7 days'",
                      con=engine)

report = Report(metrics=[DataDriftPreset(), ClassificationPreset()])
report.run(reference_data=reference, current_data=current)
report.save_html("monitoring_report.html")
```

## 자주 발생하는 오류와 해결

### `v_class_distribution_24h` 가 비어있음
- 검사_결과 의 `대표_여부=true` 행이 없음. 09번 단말 추론 INSERT 로직이 `대표_여부=true` 로 작성됐는지 확인
- 또는 INTERVAL 시간 짧음 — `INTERVAL '7 days'` 로 늘려 테스트

### 수정률이 0% — 검사원이 절대 안 고침
- AI 가 너무 정확하거나, **검사원이 그냥 그대로 저장**(검수 안 함). UX 측면에서 검사원이 진짜 검토하는지 확인 필요
- 인앱 의견 메모로 검사원에게 "AI 판정이 맞다고 생각하면 그대로, 틀리면 수정" 안내 강화

### GPU 메모리 폭증 (>14GB)
- vLLM 동시 요청 폭주 → `--gpu-memory-utilization 0.5` 로 낮추거나 `g5.xlarge` 로 인스턴스 업그레이드 (`아키텍처_명세서.md` Part 7 위험 대응)
- 분류 모델이 GPU 에 잘못 올라간 경우 — CPU 강제(`SurfaceClassifier(..., device='cpu')` → 본 단일모델 전환에서 `UnifiedClassifier`)

### Evidently 리포트가 메모리 초과
- 7일 데이터가 너무 큼. `LIMIT 100000` 으로 샘플링 또는 클래스별 1000 행으로 정규화

### 파일럿 검사원이 앱 사용을 거부
- 19번 MDM 배포 강제 정책 + 사용 안내 + 짧은 영상 가이드 필요
- UX 마찰점 발견 시 즉시 hotfix (18번 자동 빌드 + 16번 OTA 콤보로 빠른 배포 가능)
