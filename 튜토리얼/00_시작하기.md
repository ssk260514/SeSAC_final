# 00. 시작하기 — 우리가 만들 앱과 매뉴얼 사용법

> **읽는 데 필요한 시간**: 약 15분
> **이 매뉴얼이 끝나면**: Flutter 앱 + FastAPI 백엔드를 처음부터 만들어 1주차 MVP와 2주차 단말 추론까지 동작시키게 됩니다.

---

## 1. 우리가 만드는 것

**"똑똑한 품질 검사원" — 선박 LNG탱크 부품 품질 검사 AI 앱**

검사원이 안드로이드 폰의 카메라로 탱크 부품을 촬영하면, 앱이 다음 세 가지를 한 번에 알려줍니다.

1. **이게 무슨 결함인가?** (예: "균열-도장", "스크래치-모재" 등 30개 클래스 = 불량 19 + 양품 11; 단일 통합 모델)
2. **어디가 문제인가?** (Grad-CAM 히트맵으로 결함 부위를 색깔로 표시)
3. **어떻게 조치해야 하나?** (RAG가 사내 매뉴얼을 검색해서 "1) 사포 #180 으로 50mm 연마 2) 프라이머 도포..." 같은 단계별 가이드를 생성)

### 1-1. 화면 흐름 (1주차 MVP 기준)

```
[로그인] → [탱크/위치 선택] → [대시보드] ──┬─→ [카메라] → 검사 이력에 결과 추가
                                              ├─→ [검사 이력] → 정밀 분석 모달 → [결과 처리]
                                              └─→ [설정] → 로그아웃
```

총 **7개 화면 + 1개 모달(Bottom Sheet)** 입니다.

### 1-2. 시스템 구조 (개념도)

```
[안드로이드 폰 / Flutter 앱]
        │  (HTTPS, JWT 인증)
        ▼
[FastAPI 백엔드 (Docker)]
        │
        ├─→ [PostgreSQL + pgvector]  ◀── 매뉴얼 임베딩 벡터 검색
        ├─→ [AI 분류 모델 (.pth)]    ◀── 결함 분류 + Grad-CAM
        └─→ [LLM (Qwen 2.5 7B)]      ◀── 조치 가이드 자연어 생성
```

1주차에는 모든 사진을 서버로 보내고, 2주차에는 **단말(폰)에서 1차로 분류**한 다음 양품이면 서버 호출 없이 끝내는 하이브리드 구조로 발전시킵니다.

## 2. 이 매뉴얼이 다루는 범위

| 단계 | 무엇을 하는가 | 결과물 |
|---|---|---|
| **1주차 MVP** | 7화면 전체 동작 + 모든 사진을 서버로 전송하는 단순 구조 | APK 1개, 단일 통합 모델(30클래스) 동작 |
| **2주차 확장** | 단말 TFLite 추론 + 정식 RAG (BGE-M3 + pgvector + LLM) | 양품은 폰에서 자동 종결, 불량은 서버 정밀 분석 |
| **3주차 (16·17·18)** | 모델 OTA + 양품 10% 샘플 + 오프라인 큐 + CI/CD 자동 빌드 | Firebase ML 모델 무선 갱신, S3 양품 누적, GitHub Actions APK 자동 빌드 |
| **4주차 (19·20)** | 단일 모델 안정화 + MDM 배포 + 파일럿 + 모니터링 | 사내 MDM 자동 설치, JWT 자동 갱신, 단일 모델 30클래스 운영 대시보드, 파일럿 5~10명 |

3·4주차는 1·2주차가 끝나고 팀이 익숙해진 다음에 **본 매뉴얼 16~20 단계로 이어집니다**.

## 4. 만나게 될 용어 사전

처음 보면 막막한 용어들입니다. 지금은 한 줄씩만 이해하고, 실제로 코드에 등장할 때 자세히 보면 됩니다.

### 4-1. Flutter / Dart 세계

| 용어 | 한 줄 설명 |
|---|---|
| **Flutter** | 구글이 만든 크로스 플랫폼 UI 프레임워크. 같은 코드로 안드로이드·iOS 양쪽 빌드 가능 |
| **Dart** | Flutter가 쓰는 프로그래밍 언어. JavaScript와 비슷한 느낌이지만 타입이 명확 |
| **Widget** | Flutter UI의 모든 것. 버튼·텍스트·화면 전체가 다 Widget |
| **`pubspec.yaml`** | 프로젝트의 의존성·자산을 선언하는 파일. Node.js의 `package.json`에 해당 |
| **`flutter pub get`** | 의존성 다운로드 명령. `npm install`에 해당 |
| **`flutter doctor`** | 개발 환경 진단 명령. 안 풀린 게 있으면 빨간 X로 알려줌 |

### 4-2. 상태관리·라우팅

| 용어 | 한 줄 설명 |
|---|---|
| **Riverpod** | Flutter의 상태관리 라이브러리. "이 화면이 어떤 데이터를 보여줄지"를 깔끔하게 관리 |
| **Provider** | Riverpod에서 데이터를 외부로 노출하는 단위. "이 데이터 좀 줄게"의 의미 |
| **StateNotifier** | 상태(State)와 그 상태를 바꾸는 메서드를 묶어놓은 클래스 |
| **freezed** | 불변(immutable) 데이터 클래스를 자동 생성해주는 코드 생성기 |
| **go_router** | Flutter의 라우팅 라이브러리. URL → 화면 매핑을 깔끔하게 |
| **ShellRoute** | 하단 탭처럼 "여러 화면이 같은 껍데기를 공유"하는 라우팅 패턴 |

### 4-3. 네트워크·저장소

| 용어 | 한 줄 설명 |
|---|---|
| **Dio** | Flutter의 HTTP 클라이언트. axios에 해당 |
| **Interceptor** | 요청·응답을 가로채서 공통 처리(JWT 자동 추가 등)를 하는 미들웨어 |
| **JWT** | "당신이 누구인지" 증명하는 암호화된 문자열. 로그인하면 받음 |
| **Access Token / Refresh Token** | JWT의 두 종류. Access는 30분만 유효, Refresh는 7일. Access 만료되면 Refresh로 재발급 |
| **flutter_secure_storage** | JWT 같은 민감 데이터를 OS 키체인(안드로이드 Keystore)에 안전하게 저장 |
| **sqflite** | Flutter용 SQLite. 오프라인 큐 저장에 사용 |

### 4-4. 백엔드·DB

| 용어 | 한 줄 설명 |
|---|---|
| **FastAPI** | Python의 모던 웹 프레임워크. Node.js의 Express와 비슷한 위치 |
| **Uvicorn** | FastAPI 앱을 실제로 띄우는 HTTP 서버 |
| **SQLAlchemy** | Python의 ORM. SQL을 Python 객체로 다룸 |
| **PostgreSQL** | 우리가 쓰는 관계형 DB |
| **pgvector** | PostgreSQL에 벡터 검색 기능을 추가하는 확장. RAG의 핵심 |
| **bcrypt** | 비밀번호를 안전하게 해시(암호화)하는 알고리즘 |
| **Docker** | 앱·DB를 컨테이너로 격리해 어디서든 똑같이 동작시키는 도구 |

### 4-5. AI / ML

| 용어 | 한 줄 설명 |
|---|---|
| **TFLite (TensorFlow Lite)** | 모바일·임베디드용 경량 AI 모델 포맷. 단말 추론에 사용 |
| **`.tflite` / `.pth`** | 단말용(.tflite, TensorFlow) / 서버용(.pth, PyTorch) 모델 파일 |
| **MobileNetV3** | 모바일에 최적화된 이미지 분류 신경망 |
| **양자화(Quantization)** | 모델 크기 줄이기. 32비트 → 8비트로 압축 (정확도 약간 손실, 속도·크기 큰 이득) |
| **Grad-CAM** | "AI가 어디를 보고 판단했는가"를 히트맵으로 시각화하는 기법 |
| **임베딩(Embedding)** | 텍스트를 1024개 숫자(벡터)로 변환한 것. 비슷한 의미 → 비슷한 벡터 |
| **BGE-M3** | 다국어 임베딩 모델. 한국어 매뉴얼도 잘 처리 |
| **RAG (Retrieval-Augmented Generation)** | "검색 + 생성". 매뉴얼에서 유사 청크 찾아 → LLM에 넣어 답변 생성 |
| **LLM** | Large Language Model. GPT 같은 거대 언어 모델. 우리는 Qwen 2.5 7B 사용 |
| **vLLM** | LLM을 효율적으로 띄우는 추론 서버 |

## 6. 사용 규칙 (꼭 지켜야 함)

### 6-1. **명세서가 항상 우선**

- 버튼 라벨이나 토스트 문구는 `화면_명세서.md`나 `frontend/design/PAGE/0N_*/COMPONENT.md`의 **원문 그대로** 사용하세요. "로그인" → "Login"으로 임의 번역 금지.
- 색깔이나 간격 같은 디자인 토큰은 직접 HEX(`#FFA726`)를 코드에 박지 말고, 항상 **토큰 이름**(`primaryContainer`)을 사용하세요.
- API 요청/응답 JSON 키 이름은 `API_SPEC.md`의 영문 키를 그대로 따르세요 (예: `inspector_id`, `tank_type`).

### 6-2. **순서대로 진행하기**

각 파일 끝에는 다음과 같은 체크리스트가 있습니다:

```
✅ 다음 단계로 가기 전 체크리스트

- [ ] 로그인 버튼 클릭 시 토스트가 뜬다
- [ ] flutter analyze 통과
```

**모든 항목에 체크가 들어가야** 다음 파일로 넘어가세요. 한 항목이라도 안 되면 그 단계에서 멈추고 트러블슈팅하세요.

### 6-3. **막혔을 때**

1. 그 파일의 "자주 발생하는 오류와 해결" 섹션 확인
