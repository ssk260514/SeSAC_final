## 📄 1페이지 요약 (Executive Summary)

**선박 LNG탱크 부품 품질 검사 앱**("똑똑한 품질 검사원")은 검사원이 카메라로 부품을 촬영하면 AI가 즉시 결함 유형·신뢰도·조치 가이드까지 제공하는 Flutter Android 현장 보조 앱이다. 단말 1차 분류와 서버 정밀 분석을 결합한 **하이브리드 추론**으로 통신 음영 지역에서도 검사가 끊기지 않으며, RAG 기반 매뉴얼 검색으로 "다음 액션"까지 안내한다. 4주 안에 **단일 통합 모델(30클래스)** MVP에서 안정 운영까지 도달하는 것을 목표로 한다.

### 핵심 지표 6선

| 지표 | 값 | 비고 |
|---|---|---|
| 단말 추론 지연 | **100~300ms** | TFLite + NNAPI Delegate, 네트워크 무관 (`[2주차 도입]`) |
| 서버 응답 시간 | **1~2초** | 전처리 + .pth 재추론 + Grad-CAM + RAG 포함 |
| 양품 자동 종결 임계값 | **신뢰도 ≥ 0.85** | 전역 단일값 (공정별 조정 없음) |
| 모델 OTA 갱신 | **앱 재설치 0회** | Firebase ML + SHA-256 검증 (`[3주차 도입]`) |
| 세션 정책 | **1일 1세션** | partial unique index로 DB 레벨 강제 |
| 분류 클래스 | **단일 모델 30클래스** | 불량 19 + 양품 11 (6공정 결함 통합) |

### 4주 마일스톤

| 주차 | 마일스톤 | 결과물 |
|---|---|---|
| **1주차** | MVP 완성 (모든 사진 서버 전송 + 키워드 검색) | APK 1개·표면처리 .pth |
| **2주차** | 단말 추론 + 정식 RAG | `best_model_v5_datamatch_full.tflite` (≤5MB, 30클래스)·RAG 3초 이내 |
| **3주차** | 모델 OTA + 데이터 수집 + 오프라인 큐 | Firebase ML 게시·자동 빌드 |
| **4주차** | 단일 모델 안정화·MDM 배포·파일럿 | 단일 `.tflite`·파일럿 보고서 |

## 2. 핵심 가치 제안 (Value Proposition)

기존 "사진을 서버로 보내 분류 결과만 받는" 단순 구조 대비, 본 제품은 다음 **5가지 차별점**을 제공한다.

### 2-1. 하이브리드 추론 — 통신·속도·비용·보안 동시 해결

- 단말 1차 분류(100~300ms) + 서버 폴백(불량/저신뢰도)
- 100개 부품 검사 시 **30초** vs 서버 단독 200초+
- 양품은 단말에서 종결 → 서버 트래픽·도크 이미지 노출 최소화
- 통신 음영 지역(격실·이중선체)에서도 양품 판정은 즉시 완료

### 2-2. Grad-CAM 히트맵 — "왜 불량인지" 시각화

- 단순 분류 결과(`결함 유형 + 신뢰도`)를 넘어 **부위까지 색깔로 표시**
- 검사자가 육안으로 교차 검증 → AI 신뢰도 확보
- 정밀 분석 모달에서 ON/OFF 토글 + 핀치 줌 지원

### 2-3. RAG 조치 가이드 — 매뉴얼 기반 다음 액션 자동 생성

- 결함 유형 → BGE-M3 임베딩(1024차원) → pgvector Top-3 매뉴얼 청크 검색 → Qwen 2.5 7B로 조치 요약·상세 생성
- 응답 목표 시간 **3초 이내**
- Top-3 매뉴얼 출처 표기로 추적성 확보 (조치_권고_매뉴얼 N:M 매핑)
- 1주차 MVP는 **단순 키워드 검색**으로 폴백, 정식 RAG는 `[2주차 도입]`

### 2-4. 오프라인 대응 — 선박 격실에서도 끊김 없는 검사 `[3주차 도입]`

- 양품: 서버 호출 없이 즉시 완료
- 불량/저신뢰도 + 네트워크 끊김: sqflite 로컬 큐 적재, `client_request_id`(UUID v4) 멱등성 키 발급
- 네트워크 복구 시 트리거 (앱 시작 / ONLINE 전환 / 검사 이력 탭 진입)로 자동 배치 업로드 (50건 단위, 부분 실패 허용)

### 2-5. 모델 OTA — 앱 재설치 없이 모델만 갱신 `[3주차 도입]`

- Firebase ML Custom Models로 .tflite 배포
- SHA-256 해시 검증 → 검증 실패 시 기존 모델 유지
- 다음 앱 시작 시 적용 (검사 중 교체 금지)
- 사내 MDM 채널(APK)과 분리 → 모델만 자주, 앱은 드물게 갱신 가능

## 3. 사용자 시나리오 (Primary Flow + 4주차 데모)

### 3-1. Primary Flow — 양품/불량 혼합 12단계

```
①   로그인 (화면 1)
        └─ 검사원 ID + 비밀번호 → JWT 액세스(30분)/리프레시(7일) 발급, 부서 자동 조회

②   탱크/위치 선택 (화면 2)
        └─ Step 1: 탱크 유형 (B/C) → Step 2: 구역 → Step 3: 세부 위치
        └─ 선택값을 Riverpod + flutter_secure_storage에 영속화

③   세션 생성 — 1일 1세션 정책 (화면 3)
        └─ POST /api/sessions (당일 진행중 세션 있으면 409 DAILY_SESSION_EXISTS)
        └─ 409 응답 시 클라이언트는 검사 이력 탭으로 자동 이동, "이어서 검사" 라우팅
        └─ DB 레벨: uniq_active_session_per_inspector partial unique index가 race condition 차단

④   카메라 촬영 (화면 5)
        └─ 셔터 → Isolate/compute()로 백그라운드 추론, 카메라 화면 유지(연속 촬영 가능)
        └─ 결과는 StreamController로 검사 이력 카드에 비동기 반영

⑤-A 양품 + 신뢰도 ≥ 임계값
        └─ 로컬 DB 저장 + POST /api/inspect/local-result (메타만, 이미지 제외)
        └─ 10% 확률로 POST /api/inspect/sample-upload (학습 데이터 확보)

⑤-B 불량 OR 저신뢰도 (네트워크 연결됨)
        └─ POST /api/inspect (multipart): 서버 MobileNetV3(.pth) 재추론 + Grad-CAM + RAG
        └─ 검사_결과 단말 행 + 서버 행 모두 기록 (양자화 손실 추적용)

⑤-C 불량 OR 저신뢰도 (네트워크 끊김) [3주차 도입]
        └─ sqflite 로컬 큐 적재 + client_request_id 발급
        └─ 검사 이력에 '분석 대기' 카드 (회색 바)

⑥   검사 이력 (화면 4) — 카드 리스트
        └─ GET /api/sessions/{session_id}/results?status=all|완료|미완료
        └─ 빨강(불량+피드백 미저장) / 초록(양품) / 회색(미완료)

⑦   정밀 분석 모달 (Bottom Sheet)
        └─ GET /api/images/{image_id}/detail
        └─ 원본 이미지(핀치 줌) + Grad-CAM 토글 + Top-3 예측 + RAG 조치 가이드 + Source Manuals

⑧   결과 처리 (화면 6) — 검사원 피드백
        └─ 결함 유형 수정 · 심각도 · 의견 · 최종 조치 내용 입력
        └─ POST /api/results/{result_id}/feedback (UPSERT, 검사_피드백.결과_ID UNIQUE)
        └─ 조치 권고 수정은 PATCH /api/recommendations/{recommendation_id}
        └─ 검사_결과.결과_처리_상태='완료' 자동 UPDATE

⑨   세션 종료 (화면 4)
        └─ PATCH /api/sessions/{session_id}/end
        └─ 전제: 세션 내 모든 미완료 건이 0건 (없으면 400 반환, 버튼 비활성화로도 차단)

⑩   대시보드 갱신 (화면 3)
        └─ GET /api/dashboard/summary → 세션 번호·이미지 수·양품률·최근 3건 갱신

⑪   설정 (화면 7) — 푸시 알림 / 언어 토글
        └─ GET/PATCH /api/settings

⑫   로그아웃
        └─ POST /api/auth/logout (1주차 MVP: audit 로그만, 토큰 블랙리스트는 [Post-MVP])
```

### 3-2. 여정 변형 3종

| 변형 | 트리거 | 흐름 |
|---|---|---|
| **결과 재수정** | 대시보드 최근 이력 클릭 OR 검사 이력 "검사 완료" 카드 | 화면 6 진입 → 기존 피드백 표시 → "수정" 클릭 → 재입력 → 저장 (UPSERT UPDATE 분기) |

### 3-3. 4주차 데모 시나리오 (8단계 스크립트)

## 4. 기능 범위 (1주차 MVP / 2~4주차 / Post-MVP)

### 4-1. 1주차 MVP 포함 (23개 API + 7화면 + 1모달 전체)

✅ **인증**: 로그인 / 토큰 갱신 / 로그아웃 (AUTH-001~003)
✅ **탱크/공정 조회**: 탱크 유형 / 탱크-공정 매핑 / 공정 목록 (ZONE-001~002, PROC-001)
✅ **세션 관리**: 생성(1일 1세션) / 이력 / 종료 / 대시보드 요약 (SESS-001~004)
✅ **추론**: 서버 정밀 분석 (INFER-002) + 양품 단말 처리 메타 기록 (INFER-003) + 양품 10% 샘플링 (INFER-004)
✅ **모델 OTA**: 버전 조회 (OTA-001) — 활성화는 1주차 SQL 수동 실행
✅ **결과 조회**: 세션별 결과 목록 / 개별 이미지 상세 (RESULT-001~002)
✅ **피드백**: 저장(UPSERT) / 조치 권고 수정 / 세션 finalize (FEEDBACK-001~003)
✅ **설정**: 조회 / 수정 (SETTINGS-001a/b)
✅ **UI**: 7개 화면 + 정밀 분석 모달

**1주차 폴백 동작**:
- 단말 추론 미구현 → 모든 사진을 서버로 전송
- 정식 RAG 미구현 → 단순 키워드 검색으로 조치 가이드 1줄 응답
- 자동 빌드 미구현 → 수동 APK 빌드 1개

### 4-2. 단계적 도입 (2~4주차)

| 단계 | 추가 기능 | 비고 |
|---|---|---|
| **`[2주차 도입]`** | INFER-001 (단말 온디바이스 추론, TFLite + NNAPI), 정식 RAG(BGE-M3 + pgvector + Qwen 2.5 7B AWQ + vLLM), 매뉴얼 ETL | 하이브리드 분기 활성화 (양품+신뢰도≥0.85 단말 종결) |
| **`[3주차 도입]`** | INFER-005 (오프라인 배치 업로드), OTA-002 자동화 (모델 활성화 API), Firebase ML 다운로더, 양품 10% S3 비동기 업로드, 5공정 추가 학습, Codemagic/GitHub Actions 자동 빌드 | 멱등성 키 + SHA-256 검증 |

### 5-2. pgvector VECTOR(1024) 사용 위치

- **매뉴얼.내용_벡터 컬럼** — BGE-M3 dense 임베딩 표준 출력 차원(1024)과 매칭
- 검색 SQL: `ORDER BY 내용_벡터 <=> {query_vector} LIMIT 3 WHERE 공정_ID = ?`
- 공정별 매뉴얼 범위 한정(RAG 유지) + 코사인 유사도 Top-3 검색

### 5-3. 핵심 FK 관계 (텍스트 다이어그램)

```

      └─→ 매뉴얼(RAG 범위 한정)

검사원 ─┬─→ 검사_세션
        ├─→ 검사_피드백
        └─→ 사용자_설정(1:1)

검사_세션 ─→ 검사_이미지 ─→ 검사_결과 ─┬─→ 검사_피드백(1:1)
                                       └─→ 조치_권고 ─→ 조치_권고_매뉴얼 ─→ 매뉴얼
                                                       (Top-3 N:M)
```

## 9. 검증·성공 지표 (주차별)

### 9-1. 1주차 끝 (MVP)

- ☐ 안드로이드 단말에서 전체 워크플로우 동작 (로그인→탱크 선택→촬영→결과→저장)
- ☐ 표면처리 검증 사진 50장 분류 정확도 ≥ **90%**
- ☐ 라벨 검수 일치율 Cohen's κ ≥ **0.7**
- ☐ 서버 응답 1~2초 / 대시보드 로드 ≤1초
- ☐ 1일 1세션 정책 동작 (409 DAILY_SESSION_EXISTS 응답 확인)
- ☐ 미완료 건 있을 시 세션 종료 차단 (400 응답)
- ☐ JWT 액세스 30분 / 리프레시 7일 발급 + 자동 갱신 (Dio Interceptor)
- ☐ S3 업로드 + DB 저장 트랜잭션 일관성

### 9-2. 2주차 끝 (단말 추론 + 정식 RAG)

- ☐ 양품 단말 추론 ≤ **300ms** (오프라인 상태에서도 동작)
- ☐ RAG 응답 ≤ **3초** (BGE-M3 + pgvector + Qwen 2.5)
- ☐ `.tflite` 모델 파일 크기 ≤ **5MB**
- ☐ 양자화 손실 < **2%p** (TFLite INT8 vs 원본 FP32)
- ☐ RAG Top-1 매뉴얼 코사인 유사도 > **0.85**

### 9-3. 3주차 끝 (OTA + 데이터 수집)

- ☐ Firebase ML 신규 버전 게시 → 앱 재시작 시 자동 다운로드·교체·SHA-256 검증
- ☐ 양품 10% 샘플 사내 S3에 일별 누적
- ☐ 단일 모델 재학습 1회 이상 완료 (데이터 누적 반영)
- ☐ 오프라인 배치 50건 부분 실패 허용 동작 (멱등성 키 중복 처리 확인)

### 9-4. 4주차 끝 (통합·배포·안정화)

- ☐ 단일 모델 30클래스 단말·서버 양쪽 동작 + 클래스별 정확도 모니터링
- ☐ 사내 MDM에서 검사자 단말에 APK 자동 설치 검증
- ☐ GPU VRAM 사용량 ≤ **12GB** (분류 + Qwen 7B 동시 로드 시)
- ☐ 파일럿 5~10명 완료 + 피드백 수집
- ☐ JWT 자동 갱신 안정 동작 (401 → refresh → 재시도)
