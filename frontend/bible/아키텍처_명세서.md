## Part 2. 4가지 핵심 영역의 자세한 설계

### 영역 A. 단일 통합 모델 구조

**왜 단일 모델로 가는지**: 각 결함 유형당 학습 사진이 1000장 이상 충분히 확보되어, 데이터 부족·과적합 우려가 해소되었습니다. 단일 모델 1개가 30개 클래스(불량 19 + 양품 11)를 통합 분류합니다. 클래스별 데이터 불균형은 Focal Loss(포컬 로스 — 데이터가 적은 유형에 더 집중해 학습시키는 손실 함수)와 클래스 가중치로 보완합니다.

**해결책**:
- **단일 모델 1개를 만듭니다.** 30개 클래스(불량 19 + 양품 11) 전체를 한 모델이 분류합니다. 공정 구분 없이 전체 데이터를 학습하며, 유형당 1000장+ 데이터로 충분히 일반화됩니다.
- **앱 사용 흐름**: 검사자가 탱크 타입(B/C)→구역+세부위치를 선택하면 서버가 공정을 자동 결정(위치 기록·RAG 매뉴얼 범위 한정용 메타). 단말기에는 **단일 모델 파일 1개(`best_model_v5_datamatch_full.tflite`)만** 상주하며 공정과 무관하게 모든 촬영을 분류합니다 (단말기 메모리 4기가바이트 제약 여유).
- **모델 파일은 두 종류**:
  - `.pth` 파일 = PyTorch(파이토치, 페이스북이 만든 인공지능 학습용 도구) 형식. 서버에서 사용
  - `.tflite` 파일 = TensorFlow Lite(텐서플로우 라이트, 구글이 만든 단말기용 인공지능 실행 도구) 형식. 단말기에서 사용
- **데이터베이스 표 1개 유지**: `모델_레지스트리`에 **단일 모델의 버전·파일경로·해시·클래스라벨(30개)**을 기록합니다. `공정_ID`는 메타로만 유지(NULL 허용)하며 모델 선택에는 사용하지 않습니다.

**MVP(1주차)에서는**: 단일 모델 1개를 30클래스로 학습·배포합니다. 2~4주차에는 데이터 누적·라벨 정교화로 **동일 모델을 재학습**합니다 (공정별 추가 학습 개념 없음).

### 영역 B. 그래픽카드 메모리 16기가바이트(NVIDIA T4)에서 한국어 인공지능 챗봇 운영

**왜 어려운지**: Qwen 2.5 7B(70억 매개변수 한국어 인공지능 챗봇 모델)를 원본 형태(반정밀도 부동소수점, FP16)로 그대로 메모리에 올리면 약 14기가바이트를 차지합니다. 16기가바이트짜리 그래픽카드에 올리면 분류 모델·히트맵 생성 등 다른 작업이 동시에 못 돌아갑니다.

**해결책**:
- **모델을 압축**: AWQ(Activation-aware Weight Quantization, 활성화 인식 가중치 양자화 — 인공지능 모델의 숫자 정밀도를 낮춰 용량을 1/4 수준으로 줄이는 기술)를 적용한 4비트 버전을 사용합니다. HuggingFace(허깅페이스, 인공지능 모델을 무료로 공유·다운로드할 수 있는 사이트)에 공개된 `Qwen/Qwen2.5-7B-Instruct-AWQ` 파일을 그대로 사용 가능합니다. 압축 후 약 4.5기가바이트.
- **응답 빠르게 처리하는 도구**: vLLM(브이엘엘엠, 대형 인공지능 챗봇을 효율적으로 동시에 여러 사용자에게 응답시키는 오픈소스 서버 도구)를 사용. 옵션은 `--quantization awq --gpu-memory-utilization 0.6 --max-model-len 4096` (그래픽 메모리 60%까지만 사용, 한 번에 처리할 글자 길이 4096개로 제한 — 조치 가이드 답변에는 충분).
- **분류 모델은 그래픽카드 대신 CPU(중앙처리장치, 일반 컴퓨터 두뇌)에서 처리**: 단말기에서 자신감 부족으로 서버에 넘어오는 사진은 전체의 일부에 불과하므로 CPU 처리(예상 응답 1초 이내)로 충분합니다. 그래픽카드 메모리는 챗봇이 단독 사용.
- **그래픽 메모리 사용 계획**: 챗봇 9~10기가바이트 / 운영체제 여유 6기가바이트. 16기가바이트 안에 안정적으로 들어옵니다.
- **만약 메모리 부족·응답 5초 초과가 자주 발생하면**: g5.xlarge(NVIDIA A10G 그래픽카드, 메모리 24기가바이트, 시간당 약 1달러)로 인스턴스(클라우드 컴퓨터 한 대)를 업그레이드 — 4주 안에 이 상황이 오면 즉시 결정.

### 영역 C. 라벨 검수 일정

**왜 필요한지**: 인공지능은 라벨(사진에 결함 종류를 표시한 정답표)이 정확해야 잘 배웁니다. 라벨이 틀려있거나 사람마다 기준이 달라 일관성이 없으면, 학습한 인공지능도 같은 부품을 보고 다르게 판정합니다. 표면처리 13개 유형은 도장(페인트칠) / 모재(부품 본체 금속) / 보온재 위에 같은 결함이 다르게 나타나기 때문에 더 정확한 검수가 필요합니다.

**해결책 (1주차 학습 시작 전 끝내야 하는 작업)**:
- **L-M0** 라벨링 가이드 문서 작성: 13유형 각각에 "어떤 모양일 때 이 라벨인지"를 글로 적고 예시 사진 5장씩 첨부합니다. 1~2일.
- **L-M1** 표본 더블 라벨링: 무작위로 10%(50~100장)를 골라 두 명이 따로따로 라벨을 다시 답니다. 결과 비교 → 두 사람 일치율 측정 (Cohen's κ, 코헨 카파 — 두 평가자가 우연 이상으로 얼마나 일치하는지 측정하는 통계, 0.7 이상이면 신뢰 가능). 1일.
- **L-M2** 전수 교차 검수: 두 명이 데이터 절반씩 나눠 검수, 의견이 다른 5% 케이스는 세 번째 사람이 판정. 2~3일.
- **L-M3** 클래스별 데이터 수 보고서: 13유형 각각 몇 장씩 있는지 표로 정리 → 데이터가 부족한 유형은 학습 시 가중치를 다르게 주거나 데이터 증강(같은 사진을 약간 회전·밝기 조정해 사실상 데이터 양을 늘리는 기법) 강화. 0.5일.
- **검수 도구**: Label Studio(라벨 스튜디오, 오픈소스 라벨링 웹 도구) 자체 호스팅 → 라이선스 비용 0원.

### 영역 D. Firebase ML과 사내 MDM 두 채널 분리 운영

**왜 두 채널인지**:
- 사내 MDM(앞서 설명한 단말기 원격 관리 시스템)은 **앱 자체(APK 파일 = 안드로이드 앱 설치 파일)** 를 검사자 단말기에 강제로 설치합니다.
- Firebase ML은 **인공지능 모델 파일(.tflite)** 만 따로 배포해 앱을 다시 설치하지 않고도 모델만 갱신할 수 있게 합니다 (모델은 자주 바뀌지만 앱 자체 코드는 자주 안 바뀜).

**보안 분리 설계**:
- Firebase는 별도 구글 계정·구글 클라우드 프로젝트를 새로 만들어 분리 관리합니다 (회사 메인 구글 계정과 섞지 않음).
- Firebase에는 **모델 가중치 파일만** 올립니다. 매뉴얼 텍스트·검사 결과 등 다른 정보는 일체 올리지 않습니다 (회사 영업 비밀 보호).
- 라벨 매핑 파일(`metadata.json` — 모델 출력 0번이 "균열-도장", 1번이 "균열-보온재"라는 식의 변환표)은 사내 S3(Simple Storage Service, 아마존이 제공하는 파일 저장소 서비스)에만 두고 앱이 사내 서버를 거쳐 다운로드합니다.

**모델 갱신 흐름**:
1. 앱이 시작되면 사내 서버(FastAPI — 파이썬으로 짠 웹 서버 도구)에 "표면처리 모델 최신 버전이 무엇인가요?"를 묻습니다.
2. 서버가 "버전 3, 다운로드는 Firebase에서, 파일 무결성 검증용 해시값은 abc123..." 응답.
3. 앱이 Firebase ML 도구로 모델 파일을 백그라운드(앱 사용 중에도 보이지 않게 처리)로 다운로드.
4. 받은 파일의 해시(파일이 변조되지 않았는지 확인하는 지문 같은 값, SHA-256 알고리즘 사용)가 서버가 알려준 값과 같으면 다음 앱 시작 시 적용. 다르거나 다운로드 실패면 기존 모델 유지.

## Part 3. 1주차 MVP 범위 (4주 일정의 1주차 끝까지 동작해야 함)

**1주차 MVP에 들어가는 항목 (반드시 동작)**:
- [ ] 30클래스 통합 라벨 검수 완료 (영역 C의 L-M0~L-M3, 1주차 화·수까지)
- [ ] 단일 통합 모델 학습 완료 (`best_model_v5_datamatch_full.pth`, 30클래스)
- [ ] 사내 서버(FastAPI) 가장 단순한 형태: 사진을 받아 분류 결과 + 어디가 결함인지 색칠한 히트맵(Grad-CAM, 그래디언트 가중치 클래스 활성화 매핑 — 인공지능이 사진의 어느 부분을 보고 판정했는지 빨갛게 색칠해 주는 기법) + 매뉴얼 검색 결과 1줄을 한 번의 응답으로 돌려보내기
- [ ] 매우 단순한 매뉴얼 검색: 표면처리 매뉴얼 PDF 1개를 미리 텍스트로 변환해 단순 키워드 검색만 (RAG 없이) — 1주 안에 만들 수 있는 형태
- [ ] Flutter(플러터, 구글이 만든 안드로이드·아이폰 앱을 한 번에 만들 수 있는 개발 도구) 앱 화면 7개 + 1개 모달:
  - 화면 1: 로그인 화면 (검사원 ID 인증)
  - 화면 2: 탱크 유형 및 위치 선택 화면 (2단계 — 탱크 B/C 카드 → 구역+위치)
  - 화면 3: 대시보드 (3개 요약 카드 + 최근 이력)
  - 화면 4: 검사 이력 (진행 중 세션 결과 카드 리스트 + 필터)
  - 화면 4-1: 정밀 분석 모달 (Bottom Sheet — Top-3 예측 + Raw Feed + Grad-CAM + RAG 조치 가이드)
  - 화면 5: 카메라 촬영 화면 (촬영 후 카메라 유지, 백그라운드 추론)
  - 화면 6: 결과 처리 (읽기 전용 기본 → 수정 → 저장)
  - 화면 7: 설정 (프로필, 알림, 언어, 로그아웃)
  - 공통: Bottom Navigation Bar 4탭 (대시보드, 검사 이력, 결과 처리, 설정)
- [ ] 사진을 서버에 보내 결과를 받는 통신 로직
- [ ] 1주차 MVP는 **단말기 자체 추론 없이 모든 사진을 서버에 보내는 구조**로 단순화 (단말기 추론은 2~3주차에 추가)
- [ ] 첫 데모 가능한 안드로이드 설치 파일(APK) 1개 생성 (수동 빌드, 자동화는 2주차 이후)

**MVP에서 빼는 것 (2~4주차로 미룸)**:
- 단말기 자체 추론 (TensorFlow Lite)
- 모델 자동 갱신(Firebase ML)
- 진짜 RAG(사내 매뉴얼 임베딩+벡터 검색+챗봇 응답 생성)
- 양품 10% 샘플링
- 자동 배포(Codemagic·Jenkins)
- 사용자 인증 정교화(JWT 자동 갱신)
- 모니터링·피드백 루프

### 1주차 — MVP(최소 동작 시제품) 완성

**1주차 끝 시점에 동작해야 하는 시나리오**:
검사자가 안드로이드 단말기에서 앱 실행 → 로그인(검사원 ID) → 탱크 B 선택 → 구역·위치 선택(표면처리 자동 매핑) → 대시보드 확인 → "검사 시작" 클릭 → 카메라로 도장 사진 촬영(연속, 카메라 유지) → 검사 이력 탭에서 분석 결과 확인(불량 카드 → 정밀 분석 모달: Top-3 예측 + Grad-CAM + 매뉴얼 조치 가이드) → "결과 처리" 버튼 → 결과 처리 화면에서 확인/수정 → "저장" → 검사 이력으로 돌아와 "검사 완료" → 대시보드 최근 이력에서 완료된 결과 확인/재수정 가능.

### 2주차 — 단말기 자체 추론 + 정식 매뉴얼 검색

| 일정 | 작업 | 결과물 |
|---|---|---|
| 1일 | 단일 모델을 단말기용으로 변환: `.pth` → ONNX(오엔엔엑스, 인공지능 모델을 여러 프레임워크 사이에서 옮길 수 있는 표준 형식) → `.tflite`. 모델 크기 압축(INT8 양자화 — 인공지능 모델 안의 숫자를 32비트에서 8비트로 줄여 파일 크기 1/4, 속도 2~4배). 변환 후 정확도 손실 2%포인트 이하 검증 | `best_model_v5_datamatch_full.tflite` (≤5MB, 30클래스) |
| 2일 | Flutter 앱에 단말기 추론 기능 추가 — `tflite_flutter`(플러터에서 .tflite 모델을 실행하는 라이브러리) + NNAPI Delegate(엔엔에이피아이 위임 — 안드로이드의 신경망 가속 기능, 단말기 안의 인공지능 전용 칩을 활용해 추론을 5~10배 빠르게) | 단말기에서 100~300밀리초 안에 분류 |
| 3일 | 하이브리드 분기 구현: 양품 + 자신감 0.85 이상 → 단말기 결과만 사용. 그 외 → 서버 호출 | 양품은 서버 호출 없이 끝남 |
| 4일 | 진짜 RAG 추가: BGE-M3(중국 베이징지능연구원이 만든 다국어 문장 의미 임베딩 모델 — 문장을 숫자 벡터로 바꿔 의미가 비슷한 문장끼리 찾을 수 있게 함, **dense 출력 1024차원**) + pgvector(피지벡터 — PostgreSQL에 벡터 검색 기능을 추가하는 확장팩, 본 프로젝트는 `VECTOR(1024)` 사용) + Qwen 2.5 7B AWQ + vLLM 통합 | RAG 응답 3초 이내 |
| 5일 | 매뉴얼 ETL(Extract-Transform-Load — 외부 데이터를 추출·변환·적재하는 표준 데이터 처리 흐름) 1회성 스크립트: 표면처리 PDF·DOCX 전부 → 의미 단위로 자르기(Chunking) → BGE-M3 임베딩 → pgvector 적재 | 매뉴얼 검색 데이터베이스 완성 |

### Post-MVP 마이그레이션 참고 — 세션 중단 기능 재추가 시

세션 중단 기능을 도입할 때 필요한 변경 사항:

1. **DB 마이그레이션 (PostgreSQL ≥ 12)**:
   ```sql
   ALTER TYPE 세션_상태 ADD VALUE IF NOT EXISTS '중단' AFTER '완료';
   ```
   - PostgreSQL 11 이하는 트랜잭션 블록 밖에서 실행 필요
   - 한 번 추가된 ENUM 값은 제거 불가 (롤백하려면 타입 재생성)

2. **신규 엔드포인트**: `PATCH /api/sessions/{id}/abort` (SESS-005 가칭)
   - 이유 코드(`abort_reason`) + 부분 통계 보존 정책 별도 결정
   - 진행중 → 중단 전이만 허용 (완료된 세션은 abort 불가)

3. **UI 추가 위치**: 화면 4(검사 이력) 또는 화면 6(결과 처리)에 "세션 중단" 버튼 + 확인 다이얼로그

4. **API 필터 갱신**: SESS-002 `status` 필터에 '중단' 값 허용

5. **N-10 partial unique index와의 상호작용**: `WHERE 세션_상태='진행중'` 조건이므로 '중단' 세션은 인덱스 제외 — 동일일 새 세션 시작 가능 (의도된 동작)

## Part 6. 검증 방법 (이 플랜대로 만들어졌는지 확인하는 법)

### 1주차 끝 (MVP)
- [ ] 안드로이드 단말기에서 앱 실행 → 탱크 B 선택 → 구역·위치 선택(표면처리 자동 매핑) → 대시보드 → 카메라 촬영(연속, 카메라 유지) → 검사 이력에서 결과 확인 → 정밀 분석 모달(Top-3 + Grad-CAM + 조치 가이드) → 결과 처리 → 저장 → 검사 완료 → 대시보드 최근 이력 확인
- [ ] 표면처리 검증용 사진 50장에서 분류 정확도 90% 이상
- [ ] 라벨 검수 일치율 0.7 이상 (Cohen's κ)

### 2주차 끝
- [ ] 양품 사진은 네트워크가 끊긴 상태에서도 단말기에서 단독으로 결과 표시 (300밀리초 이내)
- [ ] 불량 사진은 서버에서 진짜 RAG 응답이 3초 이내 도착
- [ ] `.tflite` 모델 파일 크기 5메가바이트 이하

### 3주차 끝
- [ ] Firebase ML에 새 버전 게시 → 앱 재시작 시 자동 다운로드·교체
- [ ] 양품 사진 10% 샘플이 사내 S3에 일별 누적
- [ ] 단일 모델 재학습 1회 이상 완료 (데이터 누적 반영)

### 4주차 끝
- [ ] 단일 모델 30클래스 단말기·서버 양쪽 동작 + 클래스별 정확도 모니터링
- [ ] 사내 MDM이 검사자 단말기에 APK를 자동 설치
- [ ] 그래픽 메모리 사용량 12기가바이트 이하 (모니터링으로 확인)
- [ ] 검사자 5명 이상이 실제 사용해 본 후 피드백 수집 완료

## 부록 — 용어 사전

본문에서 처음 나올 때 풀이를 붙였지만, 잊었을 때 빨리 찾을 수 있도록 모아둡니다.

### 인공지능·머신러닝 관련
- **MobileNetV3-Large** — 구글이 만든 사진 분류용 인공지능 모델, 작은 크기와 빠른 속도가 장점
- **PyTorch (`.pth` 파일)** — 페이스북이 만든 인공지능 학습 도구, 학습 결과를 `.pth` 파일로 저장
- **TensorFlow Lite (`.tflite` 파일)** — 구글이 만든 단말기용 인공지능 실행 도구
- **ONNX (`.onnx` 파일)** — 인공지능 모델을 여러 도구 사이에서 옮길 수 있는 표준 형식
- **양자화 (Quantization, INT8, AWQ, GPTQ)** — 모델 안의 숫자 정밀도를 낮춰 파일 크기·메모리·속도를 개선하는 기법
- **NNAPI Delegate** — 안드로이드의 인공지능 전용 칩을 활용해 추론 속도를 높이는 기능
- **Grad-CAM** — 인공지능이 사진의 어느 부분을 보고 판정했는지 빨갛게 색칠해 보여주는 시각화 기법
- **이미지넷(ImageNet)** — 1400만 장의 공개 사진 데이터, 인공지능 사전 학습에 표준
- **Focal Loss** — 데이터가 적은 유형에 더 집중해 학습시키는 손실 함수
- **Stratified K-Fold** — 데이터를 K(보통 5)개로 나눠 교차 검증하되 클래스 비율을 유지하는 방법
- **Augmentation(증강)** — 같은 사진을 회전·밝기·잡음 추가로 사실상 데이터 양을 늘리는 기법
- **MLflow** — 인공지능 실험 결과·모델 버전을 기록·관리하는 오픈소스 도구
- **Cohen's κ (코헨 카파)** — 두 평가자의 일치율을 측정하는 통계, 0.7 이상이면 신뢰 가능

### 서버·인프라 관련
- **AWS (Amazon Web Services)** — 아마존이 운영하는 클라우드 서비스 모음
- **EC2 (Elastic Compute Cloud)** — AWS가 빌려주는 가상 컴퓨터
- **g4dn.xlarge / g5.xlarge** — 인공지능 작업용 EC2 사양 명칭 (g4dn은 NVIDIA T4 그래픽카드, g5는 A10G)
- **S3 (Simple Storage Service)** — AWS가 제공하는 파일 저장소
- **CloudFront** — AWS가 제공하는 콘텐츠 빠른 전송 서비스(CDN, Content Delivery Network)
- **VPC (Virtual Private Cloud)** — AWS 안에서 회사 전용 가상 네트워크
- **IAM (Identity and Access Management)** — AWS 사용자·권한 관리
- **Docker (도커)** — 프로그램을 어디서든 똑같이 돌게 포장하는 컨테이너 도구
- **Jenkins (젠킨스)** — 코드가 깃허브에 올라오면 자동으로 빌드·배포해주는 도구
- **GitHub Actions** — 깃허브가 제공하는 자동화 워크플로
- **CI/CD (Continuous Integration / Continuous Deployment)** — 코드 변경 시 자동으로 테스트·배포하는 절차

### 데이터베이스 관련
- **PostgreSQL (포스트그레스큐엘)** — 오픈소스 데이터베이스
- **pgvector** — PostgreSQL에 벡터 검색 기능을 추가하는 확장팩
- **JSONB** — PostgreSQL이 제공하는 JSON 형식 데이터 타입

### 인공지능 챗봇 관련
- **LLM (Large Language Model, 대형 언어 모델)** — ChatGPT 같은 큰 인공지능 챗봇 모델
- **sLLM (small LLM)** — 비교적 작은 크기의 자체 호스팅 가능 언어 모델
- **Qwen 2.5 7B** — 알리바바가 만든 70억 매개변수 한국어·중국어 대응 언어 모델
- **vLLM** — 대형 언어 모델을 효율적으로 동시에 여러 사용자에게 응답시키는 서버 도구
- **BGE-M3** — 중국 베이징지능연구원이 만든 다국어 문장 의미 임베딩 모델. dense 임베딩 표준 출력은 1024차원
- **임베딩 (Embedding)** — 문장을 숫자 벡터로 바꿔 의미가 비슷한 문장끼리 찾을 수 있게 함
- **RAG (Retrieval-Augmented Generation, 검색 증강 생성)** — 매뉴얼 검색 결과를 챗봇에 함께 입력해 더 정확한 답을 생성하는 기법
- **Chunking (청킹)** — 긴 문서를 의미 단위로 자르는 작업

### 앱·모바일 관련
- **Flutter (플러터)** — 구글이 만든 안드로이드·아이폰 앱을 한 번에 만들 수 있는 개발 도구
- **APK (Android Package Kit)** — 안드로이드 앱 설치 파일
- **AAB (Android App Bundle)** — 구글 플레이용 안드로이드 앱 번들 형식
- **IPA (iPhone Application Archive)** — 아이폰 앱 설치 파일
- **MDM (Mobile Device Management)** — 회사가 직원 단말기를 원격 관리·앱 설치 강제하는 시스템
- **OTA (Over-The-Air)** — 무선을 통한 자동 갱신
- **Firebase ML Custom Models** — 구글이 제공하는 인공지능 모델 배포 클라우드 서비스
- **Codemagic** — Flutter 앱 자동 빌드 서비스
- **NNAPI (Neural Networks API)** — 안드로이드의 신경망 가속 표준 인터페이스
- **NPU (Neural Processing Unit)** — 단말기 안에 들어 있는 인공지능 전용 칩
- **JWT (JSON Web Token)** — 사용자가 로그인했음을 증명하는 토큰
- **SHA-256** — 파일이 변조되지 않았는지 확인하는 해시 알고리즘

### Flutter 패키지
- **camera** — 카메라로 사진·영상 촬영
- **dio** — 네트워크 요청(통신) 라이브러리
- **flutter_secure_storage** — 토큰 등 민감 정보 안전 저장
- **sqflite** — 단말기 안의 작은 데이터베이스(SQLite) 사용
- **tflite_flutter** — 플러터에서 `.tflite` 모델을 실행
- **Riverpod / Provider** — 앱 안의 데이터 흐름을 정리하는 상태 관리 도구
- **Stack 위젯** — 화면 요소를 겹쳐 쌓는 레이아웃 (사진 위 히트맵 겹치기에 사용)

### 일반 용어
- **MVP (Minimum Viable Product)** — 정식 출시 전 핵심 기능만 동작하는 시제품
- **ETL (Extract-Transform-Load)** — 외부 데이터를 추출·변환·적재하는 데이터 처리 흐름
- **CPU (Central Processing Unit)** — 중앙처리장치, 일반 컴퓨터 두뇌
- **GPU (Graphics Processing Unit)** — 그래픽카드, 인공지능 계산도 빠르게 처리
- **RAM (Random Access Memory)** — 컴퓨터·단말기 작업 메모리
- **ARM64** — 요즘 스마트폰 두뇌 칩의 표준 종류
- **Hash (해시)** — 파일 내용을 짧은 지문 같은 숫자열로 변환한 값. 변조 확인용
- **BackgroundTasks (백그라운드 작업)** — 사용자 응답을 막지 않고 뒤에서 처리되는 작업
